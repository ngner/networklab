The Three VM Personas

Every network decision maps to one of these

PersonaDescriptionBest Attachment
Normal VM Simple IP on one or more VLANs. No trunk, no 802.1Q tags. Linux-bridge NAD or CUDN Localnet
VPC-like Tenant Namespace isolation, NetworkPolicy, persistent IPs, live migration. Primary UDN Layer2 or CUDN Localnet
Network Appliance Firewall/router expecting a raw 802.1Q trunk NIC with VLAN sub-interfaces. Linux-bridge trunk NAD ("vlan": 0)
Ask "what kind of VM?" first. The persona determines the mechanism — not the other way around.

Phase 1d/1e

macvlan vs ipvlan

macvlan

Each sub-interface gets its own MAC. Switch sees N MACs per port.
Cannot reach parent interface.

ipvlan

All sub-interfaces share parent's MAC. Switch sees 1 MAC.
L3 mode: no ARP, no broadcast.
Both are niche for KubeVirt. Cannot talk to host. Use linux-bridge NAD or CUDN Localnet instead for most VM workloads.

What OVS CNI Used to Do — and Why It's Gone

OVS CNI (removed OCP 4.17+)

"type": "ovs" NADs created ports on OVS bridges.

Per-VM "trunk": [{"id": 100}] allowed VLAN filtering.

Conflicts with OVN-Kubernetes — OVN manages all OVS bridges. Binary removed.

failed to find plugin "ovs" in path [/var/lib/cni/bin]

Replacements on OCP 4.17+

Linux-bridge trunk NAD ("vlan": 0) — all VLANs, no per-VM filtering.

CUDN Localnet — OVN-managed per-VLAN access. NetworkPolicy enforced.

SR-IOV — hardware VLAN filtering per VF.

Phase 4

OVN Manages the OVS Bridge Now

On OCP 4.17+, OVN-Kubernetes exclusively manages all OVS bridges via the southbound database. You do not create ports manually with ovs-vsctl add-port, and you do not use the ovs CNI plugin.

Phase 2–3 (host-level OVS): You manipulated OVS directly — ovs-vsctl, flow tables, trunk lists.

Phase 5 (OCP): OVN compiles Kubernetes resources (CUDN, UDN, NetworkPolicy) into OVS flows automatically. Your debug tools still work: ofproto/trace, ovn-trace.

Phase 5

Linux-bridge NAD — When and How

Simplest physical underlay access. No NetworkPolicy. Good for appliances and simple VMs.

Access NAD ("vlan": 100)

Normal VM gets an untagged NIC on VLAN 100.

Bridge strips/adds VLAN tag automatically.

Trunk NAD ("vlan": 0)

Appliance gets 802.1Q tagged frames on all VLANs.

No per-VM VLAN filtering — all VMs see all VLANs.

No IPAM for VMs on linux-bridge NADs. Use cloud-init.
Cannot attach to the default machine network directly.
Lab: Section 5c
Phase 5

CUDN Localnet — When and How

OVN-managed physical underlay. NetworkPolicy. Per-VLAN isolation.

API: ClusterUserDefinedNetwork (cluster-scoped). Not UserDefinedNetwork.

VLAN: Access-mode only — vlan: {mode: Access, access: {id: N}}. No 802.1Q trunk.

Needs: OVS bridge (ovs-br1) + OVN bridge-mapping via NMState NNCP.

Scope: namespaceSelector controls which namespaces consume the network.

CUDN Localnet IPAM for VMs must be Disabled.
MultiNetworkPolicy on secondaries: ipBlock only — pod/namespace selectors not supported.
Lab: Section 5c
Phase 5

UDN Layer2 — VPC-like Tenant Isolation

No physical NIC needed. Persistent IPs. Live migration. Built-in NetworkPolicy.

API: UserDefinedNetwork (namespace-scoped) or ClusterUserDefinedNetwork

Role: Primary (replaces default pod network) or Secondary

Topology: Geneve overlay — no VLAN config, no physical NIC

Primary UDN requires k8s.ovn.org/primary-user-defined-network namespace label at creation. Cannot be added later.
Primary UDN VMs lose virtctl ssh, oc port-forward, and headless services.
Lab: Section 5c
Phase 5

SR-IOV — Max Performance, No Migration

Hardware VF passthrough. Per-VF VLAN filtering replaces removed OVS CNI trunks.

Performance: Best — bypasses kernel and OVS entirely.

VLAN: Per-VF hardware filtering — true trunk with allow-list per VM.

Use case: Telco/NFV, DPDK workloads, or when per-VM trunk filtering is required.

No live migration (uses vfio-pci — PCI passthrough).
No hot-unplug for SR-IOV interfaces.
Lab: Section 5c
Phase 5

Per-Persona Recommendations

Normal VM

Default: CUDN Localnet (OVN policy)

Simple: Linux-bridge NAD (no policy needed)

Best perf: SR-IOV (no migration)

VPC Tenant

Default: Primary UDN Layer2 (persistent IPs)

Physical access: + CUDN Localnet per VLAN

Segmentation: NetworkPolicy on CUDN/UDN

Network Appliance

Default: Linux-bridge trunk ("vlan": 0)

Multi-NIC: CUDN Localnet per VLAN

HW filter: SR-IOV (replaces OVS CNI)

Lab: Section 5d
Phase 5

The Two-Bridge Hybrid Model

Appliances on br-secondary (NIC 2), tenants on ovs-br1 (NIC 3). Both connect to the same upstream — cross-VLAN traffic flows through the firewall.
Lab: Section 5e
Phase 5

Critical Constraints

1. OVS CNI removed in OCP 4.17+. failed to find plugin "ovs".
2. Localnet = ClusterUserDefinedNetwork only. UDN supports Layer2/Layer3 only.
3. CUDN Localnet VLAN is access-mode only. No 802.1Q trunk.
4. Primary UDN namespace label required at creation. Cannot be added later.
5. Primary UDN VMs lose virtctl ssh, oc port-forward, headless services.
6. No IPAM for VMs on linux-bridge NADs or CUDN Localnet. Use cloud-init.
7. SR-IOV: no live migration, no hot-unplug.
8. Hot-plug limits: 4 PCI slots total (including existing interfaces).
9. Default pod network traffic interrupted during live migration.
10. MultiNetworkPolicy on secondaries: ipBlock only for VMs.

What Was Removed & Why

OVS CNI obituary — OCP 4.17+

Old (OVS CNI)Replacement
"type": "ovs" with "trunk": [...] Linux-bridge trunk ("vlan": 0) — no per-VM filtering
"type": "ovs" with "vlan": N CUDN Localnet with vlan.mode: Access
Per-port VLAN allow-lists SR-IOV with VF VLAN filtering (hardware)
Why removed: OVN-Kubernetes manages all OVS bridges via the southbound database. A standalone CNI that directly manipulates OVS ports conflicts with OVN's control plane — ports get deleted or reconfigured silently.
