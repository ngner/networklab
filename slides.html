Phase 1d/1e

macvlan vs ipvlan

macvlan

Each sub-interface gets its own MAC. Switch sees N MACs per physical port.
Cannot reach parent interface.

ipvlan

All sub-interfaces share parent's MAC. Switch sees 1 MAC.
L3 mode: no ARP, no broadcast.
Choose macvlan for L2 access (unique MACs). Choose ipvlan when the switch has MAC-per-port limits or you want L3-only isolation at scale.

Per-VLAN Bridge

1 Create br-300
2 Create veth pair
3 Move into namespace
4 Assign IP, bring up
5 Reconfigure guest

Disruptive. Guest sees a new NIC.

VLAN-Aware Bridge

1 bridge vlan add vid 300 dev veth-dmz-br

Live, non-disruptive.

⚠ Not persisted across reboot.

OVS

1 ovs-vsctl set port ... trunks=100,200,300

Live, non-disruptive.

✓ Persisted in OVSDB.

Guest-side: all three need ip link add ... type vlan id 300 inside the namespace — but only the per-bridge model forces a new host-side NIC.

Phase 3

Read-Only Monitor Port

Linux Bridge

Requires a separate subsystem (nftables):

nft add table bridge filter
nft add chain bridge filter forward \
  '{ type filter hook forward priority 0; }'
nft add rule bridge filter forward \
  ibrname "veth-mon-br" drop

Bridge has no concept of port direction.
SPAN/mirror: tc mirred — fragile, separate subsystem.

OVS

Same flow table as forwarding — one command:

# Drop ingress from monitor
ovs-ofctl add-flow ovsbr0 \
  "priority=100,in_port=veth-mon-br,actions=drop"

# Or: proper SPAN mirror
ovs-vsctl -- --id=@p get port veth-mon-br \
  -- --id=@m create mirror name=all-vlans \
     select-vlan=100,123,150,200 \
     output-port=@p

Native, declarative, verifiable with ofproto/trace.

Phase 3

Proving VLAN Isolation

Can fw-dmz (VLANs 100,200) reach fw-internal (VLANs 123,150)?

Linux Bridge

# Run tcpdump and look for absence of traffic
ip netns exec fw-internal tcpdump -i veth-int &
ip netns exec fw-dmz ping -c 3 10.0.123.1
# tcpdump shows nothing — but…

"Absence of evidence is not evidence of absence"

OVS

# Prove without generating traffic
ovs-appctl ofproto/trace ovsbr0 \
  "in_port=veth-dmz-br,dl_vlan=123,\
   dl_src=aa:bb:cc:dd:ee:01,\
   dl_dst=ff:ff:ff:ff:ff:ff"

# Output: "Datapath actions: drop"

Deterministic proof. Full pipeline trace.

ofproto/trace lets you prove security properties without generating traffic — critical for compliance and auditing.

Phase 3f

Performance: What the Numbers Say

Datapath architecture drives the differences

VLAN-Aware Bridge

Pure kernel fast path. VLAN classification is a bitmap — zero per-packet cost regardless of VLAN count. No userspace component.

OVS

Two-tier. Cached flows are fast; cache misses trigger userspace upcall. Latency is bimodal.

Phase 3f

Performance → Decision Impact

At typical firewall VM scale (< 50 VMs, < 100 VLANs), throughput and latency differences are negligible. Choose based on operability, not performance.
Workload profile Performance says…
Firewall VM, typical enterprise traffic Gap is noise. Pick OVS for operability.
High PPS, small packets (telco, NFV) Bridge wins 5–15% PPS. Consider SR-IOV.
Latency-sensitive (finance, HFT) P99.9 tail matters. Bridge or SR-IOV.
100+ VLANs, 1000+ flows Monitor OVS megaflow cache & upcall rate.
Minimal resource budget Bridge: zero userspace daemons, kernel slab only.
Median is close, tails diverge. OVS latency is bimodal — cached flows are fast, megaflow misses produce spikes. The performance gap only becomes a factor at very high PPS or when P99.9 latency is a hard requirement.

Phase 5

Secondary Networks: Multus vs UDN

Multus + Bridge/OVS CNI

How: NetworkAttachmentDefinition → CNI plugin creates port on node bridge

VLAN: "vlan": 100 for access, "vlan": 0 for trunk

IPAM: External (Whereabouts / static)

NetworkPolicy: Does not apply to secondary interfaces

Scope: Node-local bridge (L2 within node)

UDN (User Defined Network)

How: OVN logical switch → managed by OVN-Kubernetes

Topologies: Layer2 / Layer3 / Localnet

IPAM: Built-in (OVN DHCP)

NetworkPolicy: Applies natively (OVN ACLs)

Scope: Cluster-wide (Geneve overlay or localnet)

UDN is the future direction on OVN-Kubernetes. Use Multus bridge CNI for raw L2/trunk pass-through; use UDN for managed, policy-aware networks.

Phase 5

Three Approaches to Firewall VM Trunks on OCP

Bridge CNI trunk

"vlan": 0

VM sees: All VLANs on one trunk NIC
Per-VM filter: No
Day-2 add VLAN: N/A (all VLANs pass)
NetworkPolicy: No
ofproto/trace: No (linux bridge)

Simple. All VMs on the bridge see the same VLANs.

OVS CNI trunk

"trunk": [{"id": N}]

VM sees: One trunk NIC, tagged 802.1Q
Per-VM filter: Yes (per-NAD)
Day-2 add VLAN: Edit NAD, no restart
NetworkPolicy: No (bypasses OVN)
ofproto/trace: Yes

Best for firewall appliances expecting 802.1Q trunks.

Localnet per-VLAN

One UDN per VLAN

VM sees: One NIC per VLAN, untagged
Per-VM filter: Yes (which UDNs attached)
Day-2 add VLAN: New UDN + VM spec, restart
NetworkPolicy: Yes (OVN enforces)
ofproto/trace: Yes (OVN flows)

Best when VM doesn't need tags or you want per-VLAN policy.

For firewall VMs expecting a standard 802.1Q trunk, OVS CNI is the right choice. Localnet-per-VLAN is better when you want OVN NetworkPolicy enforcement on each VLAN individually.
