Phase 1d/1e

macvlan vs ipvlan

macvlan

Each sub-interface gets its own MAC. Switch sees N MACs per physical port.
Cannot reach parent interface.

ipvlan

All sub-interfaces share parent's MAC. Switch sees 1 MAC.
L3 mode: no ARP, no broadcast.
Choose macvlan for L2 access (unique MACs). Choose ipvlan when the switch has MAC-per-port limits or you want L3-only isolation at scale.

OVS & OVN Background

You don't configure OVS directly on OpenShift

Phase 2

OVS Is the OVN-Kubernetes Datapath

OVN-Kubernetes owns the OVS bridges on every OCP node:

  • br-int — integration bridge (pod networking, NetworkPolicy, services)
  • br-ex — external bridge (node-to-external traffic)

For this workshop, two things matter:

Bridge CNI vlanTrunk

Uses a Linux bridge (kernel). Does not touch OVS at all. Same construct from Phase 1.

Localnet CUDN

Uses an OVS bridge configured via NMState bridge-mappings. OVN programs the flows.

For debugging OVN-programmed flows on a node:

oc debug node/<node> -- chroot /host ovs-ofctl dump-flows br-int
oc debug node/<node> -- chroot /host ovs-vsctl show

Lab Exercises

Exercise 2 — Day-2 VLAN change:

bridge vlan add vid 300 dev veth-dmz-br

Live but not persisted across reboot. On OpenShift, NMState + Bridge CNI NADs handle persistence declaratively.

Exercise 3 — Prove VLAN isolation:

fw-dmz sends VLAN 123 (not in its allow-list) → bridge drops it. fw-internal never sees the frame.

Exercise 5 — Asymmetric expansion:

bridge vlan add vid 100 dev veth-int-br

Gives fw-internal access to VLAN 100 in addition to 123/150 — a single command. With the per-bridge model this would mean creating a new veth pair and reconfiguring the guest.

The VLAN-aware bridge with per-port filtering is the exact kernel primitive behind the vlanTrunk parameter in the bridge CNI plugin on OpenShift.

What Is a User Defined Network?

Phase 5e

UDN vs CUDN: Scope

Namespace-scoped vs cluster-wide — like a single-account VPC vs a shared VPC

Use UDN for single-namespace isolation. Use CUDN when multiple namespaces share the same L2 segment — like sharing a VPC via AWS RAM.
Lab guide: Section 5e
Phase 5e

UDN Topologies — Three Ways to Wire a Network

Layer2

AWS analogy: single subnet in one AZ

Flat L2, same subnet. Supports live migration. Use for: KubeVirt VMs needing L2.

Layer3

AWS analogy: VPC with subnets across AZs

Per-node subnet, OVN routes. Scales like VPC across AZs. Use for: large-scale pod isolation.

Localnet

AWS analogy: Direct Connect / VPN Gateway

Bridges OVN to a physical VLAN. Access mode only (1 VLAN per CUDN).

Localnet UDN: no trunk. vlan.mode = [Access] only. For 802.1Q trunk → Bridge CNI vlanTrunk.

Phase 5

Managed Virtual Network vs Raw L2 Pipe

UDN gives you a VPC. Bridge CNI gives you a cable.

UDN / CUDN (Managed — "VPC")

How: OVN logical switch → OVS bridge

Policy: Enforced natively • Topologies: L2, L3, Localnet

Think: "VPC with subnets, security groups, and a router."

Bridge CNI (Unmanaged — "Cable")

How: NAD → bridge CNI → Linux bridge port

Policy: None (bypasses OVN) • VLAN: vlanTrunk

Think: "Raw cable with VLAN filtering to physical network."

In Model 4 you use both: UDN for internal managed domain (VPC), Bridge CNI for external raw trunk (Direct Connect cable).

Phase 5

OCP 4.21: Two Approaches to Firewall VM VLANs

Bridge CNI vlanTrunk

VM sees: One trunk NIC, tagged 802.1Q
Per-VM filter: Yes (per-NAD vlanTrunk array)
Day-2 add VLAN: Edit NAD + restart VM
NetworkPolicy: No (bypasses OVN)
Bridge type: Linux bridge

Best for firewall appliances expecting 802.1Q trunks.

Localnet CUDN (Access)

VM sees: One NIC per VLAN, untagged
Per-VM filter: Yes (which CUDNs attached)
Day-2 add VLAN: New CUDN + restart VM
NetworkPolicy: Yes (OVN enforces per-CUDN)
Bridge type: OVS bridge

Best when VM supports multi-NIC or you need per-VLAN policy.

The two approaches use different bridge types on the node. A physical NIC can only belong to one bridge — the hybrid model requires two NICs.

Model 1: All Localnet CUDN

Single OVS bridge, one NIC — firewall gets discrete NICs per VLAN (simple fallback)

Works if the firewall supports multiple discrete NICs (pfSense, OPNsense, VyOS). One bridge type, full NetworkPolicy everywhere.

Model 3: Hybrid (Two NICs Required)

Linux bridge for 802.1Q trunk + OVS bridge for Localnet CUDN

A physical NIC can only belong to one bridge (Red Hat KB 7107272). Traffic between firewall and tenant VMs flows through the physical switch.
Lab guide: Model 3

Model 4: Internal / External Boundary

Hybrid: Localnet CUDN + Bridge CNI Trunk

Primary Localnet CUDN (internal) + Bridge CNI vlanTrunk (external) — best with few internal VLANs

Lab guide: Model 4

Model 4: Strengths & Limitations

Internal Domain (OVS + Localnet CUDN)

  • Bond A → OVS bridge → Localnet CUDNs
  • Firewall gets primary UDN — first-class OVN citizen
  • Tenants get simple untagged NICs per VLAN
  • NetworkPolicy enforced on all internal traffic

External Domain (Linux bridge + Bridge CNI)

  • Bond B → Linux bridge → vlanTrunk NAD
  • Only the firewall has an interface here
  • Standard 802.1Q trunk to physical network
  • Tenants never touch the external bridge
Scalability limit: Each internal VLAN = one NIC on the firewall VM (Localnet Access = one VLAN per CUDN). Works for 2–5 VLANs; for 10+, use Bridge CNI trunk (Model 2/3).
Lab guide: Model 4

Model 4: Firewall VM Configuration

Primary Localnet CUDN + secondary CUDNs + external trunk

apiVersion: kubevirt.io/v1
kind: VirtualMachine
metadata:
  name: fw-dmz
  namespace: firewall-vms   # labelled: k8s.ovn.org/primary-user-defined-network
spec:
  template:
    spec:
      domain:
        devices:
          interfaces:
            - name: internal-100    # primary UDN (replaces cluster default)
              bridge: {}
            - name: internal-200    # secondary Localnet CUDN
              bridge: {}
            - name: external-trunk  # Bridge CNI vlanTrunk
              bridge: {}
      networks:
        - name: internal-100
          pod: {}                        # ← primary network = cudn-internal-100
        - name: internal-200
          multus:
            networkName: cudn-internal-200
        - name: external-trunk
          multus:
            networkName: firewall-vms/trunk-external

eth0 — Internal

VLAN 100 untag • Primary UDN

eth1 — Internal

VLAN 200 untag • Secondary CUDN

eth2 — External

Key Learnings

ovs-cni unsupported on OCP 4.14+

Binary does not exist on nodes. Use Bridge CNI vlanTrunk instead.

Localnet UDN = Access only (no trunk)

VLANMode enum is [Access]. Virt docs Table 10.1: "L2 trunk: localnet = No."

Two bridge types = two NICs

Bridge CNI needs Linux bridge. Localnet needs OVS bridge. Models 3 & 4 need two NICs/bonds.

Localnet = one NIC per VLAN

FW on 5 VLANs needs 5 NICs. Model 4 suits small scale; 10+ VLANs → use Bridge CNI trunk (Model 2/3).
